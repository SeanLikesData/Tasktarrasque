# Tasktarrasque Code Audit

Date: 2026-06-09

Scope: local code, build scripts, tests, documentation, persistence behavior, and SwiftUI/AppKit user interface behavior in this repository.

Baseline verification before fixes:

- `./run-tests.sh` passed.
- `./build.sh` passed.
- The built app was copied to `/Applications/Tasktarrasque.app` and launched, matching project memory for how the app is tested.

## Findings And Recommendations

### 1. Keyboard shortcuts can conflict with text editing

The main view and template sheet bind single-letter shortcuts such as `n`, `w`, `h`, `r`, and `d` at parent view level. When a task title text field is in rename mode, those handlers can still be eligible to run depending on focus routing. The risky behavior is that typing ordinary letters while editing a title may create new tasks, toggle a task, start another rename, or delete the focused task.

Recommendation: when any rename field is active, only let Return finish editing and Escape leave rename mode. Ignore task-management shortcuts, delete commands, and movement commands until rename mode ends.

### 2. Day tabs do not show the documented completion score

The README says each day tab shows a completion score such as `3/5`, but the current tab only shows the completed count next to a checkmark. This is a user-interface inconsistency and makes it harder to tell whether a day has no tasks or many incomplete tasks.

Recommendation: show each day tab's `scoreText` in the tab and update accessibility labels to include the full score.

### 3. Focus can become stale after changing weeks or days

Keyboard focus is stored as task identifiers. When the selected week or selected day changes, the focused identifier may refer to an item that is no longer visible. Subsequent keyboard actions may silently do nothing or operate from a confusing state.

Recommendation: validate focus whenever the selected week or selected day changes. Clear rename state and move focus to the first visible item only when needed.

### 4. Non-toggleable cards still look like they have a check control

`SharedTaskCard` always renders a check button, even when `onToggle` is `nil`. It is disabled, but the visual language still implies the item can be marked done. This affects This Week tasks and template items.

Recommendation: render a decorative, accessibility-hidden icon placeholder for non-toggleable cards. Keep alignment stable, but do not expose a disabled "Mark done" control.

### 5. Template sheet layout is too cramped at smaller popover sizes

The template sheet gives the left column a fixed width of 300 points. In the small popover size, that leaves too little room for the right column and its seven day tabs. The result is high text-compression pressure and inconsistent density compared with the main view.

Recommendation: reduce the fixed template left-column width and make template day tabs more compact with explicit text scaling.

### 6. Sheets do not consistently claim the full popover surface

Settings, shortcuts, and template overlays rely on implicit layout behavior. Some sheets use spacers, but they do not all explicitly fill the available width and height. This can make backgrounds or hit areas inconsistent when the popover size changes.

Recommendation: give sheets an explicit `maxWidth` and `maxHeight` frame so each sheet behaves like a full overlay.

### 7. Persistence load failures can overwrite bad data without preserving it

If `weeks.json` exists but cannot be decoded, the store reports an error, then creates a new week because the in-memory week list is empty. The next save can overwrite the unreadable file. That removes the user's best chance to recover manually from a corrupt or future-version save file.

Recommendation: when loading fails, move the unreadable file to a timestamped backup before creating new data. Keep the user-facing error clear enough to locate the backup.

### 8. Week decoding is stricter than the app's migration posture

`DayPlan` and `WeeklyTemplate` have custom decoding that tolerates missing older fields, but `WeekPlan` relies on synthesized decoding. Missing `id`, `thisWeekTasks`, `bigThree`, or `days` fields can fail the entire data load even though reasonable defaults exist.

Recommendation: add a custom `WeekPlan` decoder that preserves required `startDate` behavior while defaulting and normalizing optional historical fields.

### 9. Directory creation errors are ignored

`TaskStore` calls `createDirectory` with `try?`, so a failure is silently deferred until save time. That makes startup diagnostics less direct.

Recommendation: handle directory creation errors explicitly and set the persistence error immediately.

### 10. Reveal-in-Finder can fail when the data file does not exist

The Settings action always asks Finder to select `weeks.json`. If the file is missing, Finder may not show anything useful.

Recommendation: if the data file does not exist yet, open or reveal the containing Application Support directory instead.

### 11. Build source discovery is not deterministic or robust to spaces

`build.sh` assigns `SOURCES=$(find ...)` and expands it unquoted. This works for the current paths, but the compile order is not stable and paths with spaces would break. The tests already use a source array.

Recommendation: use a sorted Bash array for source files in `build.sh`.

### 12. Code signing failures are hidden

`build.sh` suppresses all output and ignores signing failure. Ad-hoc signing should not necessarily block a local personal build, but a hidden failure makes deployment issues harder to diagnose.

Recommendation: keep the non-blocking behavior but print a warning when signing fails.

### 13. Local deployment workflow is split between memory and scripts

The README says to open `build/Tasktarrasque.app`, while project memory says the user tests from `/Applications`. This creates an easy path to testing the wrong app copy.

Recommendation: add an install-and-launch option to `build.sh`, update README instructions, and keep the plain build path available.

### 14. Stale ignored build artifact exists locally

The ignored `build/` directory contains a stale `Notebloat.zip`, unrelated to Tasktarrasque. It is not tracked, but it is misleading during local inspection.

Recommendation: remove the stale local artifact.

### 15. Style declarations are compressed into long single-line expressions

The shared visual style file contains several long one-line view declarations. This is not a runtime defect, but it makes the UI system harder to review and easier to change inconsistently.

Recommendation: expand shared style declarations into readable multi-line definitions without changing visual behavior.

## Implementation Plan

1. Fix persistence and migration safety, then add model tests for backup and permissive decoding.
2. Fix keyboard editing guards and stale focus handling in the main view and template sheet.
3. Tighten UI consistency: day scores, non-toggleable card rendering, sheet sizing, template compact layout, error banner readability, and style formatting.
4. Fix build/deployment scripts and README.
5. Remove stale local build artifacts.
6. Run tests and build, then install and launch the final `/Applications` copy.
